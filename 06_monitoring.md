## 모니터링 영역 구분

```
클라이언트 모니터링    → 앱 · 웹에서 사용자 측 문제 감지
서버 모니터링          → 서버 · 인프라 · API 문제 감지
```

---

## 클라이언트 모니터링

### 툴 구성

| 대상 | 툴 | 용도 |
|------|-----|------|
| **앱 (iOS/Android)** | Firebase Crashlytics | 크래시 감지 · 리포트 |
| | Firebase Analytics | 사용자 행동 분석 |
| | Firebase Performance | 앱 성능 모니터링 |
| **웹 (프론트엔드)** | Sentry | JS 에러 · 성능 모니터링 |

### 툴 역할 구분

```
Firebase   → 앱 전용
Sentry     → 웹 · 서버 (앱은 Firebase와 중복될 수 있어 웹 · 서버에 집중)
```

---

## 서버 모니터링

### 툴 구성

| 대상 | 툴 | 용도 |
|------|-----|------|
| **서버 · 인프라** | Grafana + Prometheus | CPU · 메모리 · 트래픽 |
| **로그 관리** | ELK Stack | 로그 수집 · 분석 |
| **API 에러** | Sentry | 백엔드 에러 추적 |
| **업타임** | UptimeRobot | 서비스 가용성 체크 |
| **음성인식 모델** | Grafana | 모델 성능 · 지연시간 |

---

### Grafana + Prometheus 모니터링 항목

| 분류 | 모니터링 항목 | 임계값 |
|------|--------------|--------|
| **서버** | CPU 사용률 | 80% 이상 경고 |
| | 메모리 사용률 | 85% 이상 경고 |
| | 디스크 사용률 | 90% 이상 경고 |
| **API** | 응답시간 | 1초 이상 경고 |
| | 에러율 | 1% 이상 경고 |
| | 초당 요청수 (RPS) | 급격한 변화 감지 |
| **음성인식 엔진** | 모델 응답시간 | 3초 이상 경고 |
| | 인식 정확도 | 기준치 이하 경고 |
| | GPU 사용률 | 90% 이상 경고 |
| **DB** | 쿼리 응답시간 | 500ms 이상 경고 |
| | 커넥션 수 | 최대치 80% 경고 |

### ELK Stack 로그 관리

```
각 서버 · 서비스 로그
      ↓
Logstash (로그 수집 · 가공)
      ↓
Elasticsearch (로그 저장 · 색인)
      ↓
Kibana (로그 시각화 · 검색)
```

**로그 레벨 정의**

| 레벨 | 용도 | 예시 |
|------|------|------|
| **ERROR** | 즉시 대응 필요 | API 500 에러 |
| **WARN** | 주의 필요 | 응답시간 지연 |
| **INFO** | 일반 정보 | 배포 완료 |
| **DEBUG** | 개발 디버깅용 | 상세 실행 흐름 |

**환경별 로그 수집 규칙**

```
Production  → ERROR · WARN · INFO만 수집
Staging     → 전체 레벨 수집
Dev         → 전체 레벨 수집
```

### UptimeRobot (업타임 모니터링)

```
주요 엔드포인트를 1분마다 체크
      ↓
응답 없음 감지 시
      ↓
즉시 Slack #deploy-production 알림
```

## 장애 대응 프로세스

### 장애 등급 정의

| 등급 | 기준 | 대응 시간 |
|------|------|----------|
| **P1 (Critical)** | Production 전체 서비스 중단 | 즉시 (15분 내) |
| **P2 (High)** | 핵심 기능 장애 (로그인 · 결제 등) | 30분 내 |
| **P3 (Medium)** | 일부 기능 장애 | 2시간 내 |
| **P4 (Low)** | 경미한 오류 · UI 버그 | 다음 스프린트 |

---

### 장애 대응 흐름

```
모니터링 툴 이상 감지
      ↓
Slack #deploy-production 자동 알림
      ↓
DevOps / 온콜 담당자 확인 (15분 내)
      ↓
장애 등급 판단 (P1~P4)
      ↓
┌─────────────────────────────┐
│ P1 / P2                     │
│ → 즉시 관련 팀 리드 호출   │
│ → 원인 파악                 │
│ → Hotfix or 롤백 결정       │
└─────────────────────────────┘
      ↓
┌─────────────────────────────┐
│ 롤백이 빠른 경우            │
│ → 이전 버전으로 즉시 롤백   │
│                             │
│ 수정이 빠른 경우            │
│ → Hotfix 브랜치 생성        │
│ → 긴급 수정 후 배포         │
└─────────────────────────────┘
      ↓
서비스 정상화 확인
      ↓
Post-mortem 작성 → Confluence 문서화
```

---

### 온콜 (On-call) 체계

| 시간대 | 대응 방식 |
|--------|-----------|
| **평일 업무시간** | 담당 팀 즉시 대응 |
| **평일 업무 외** | 온콜 담당자 Slack 알림 + 전화 |
| **주말** | 온콜 담당자 Slack 알림 + 전화 |

```
온콜 담당자 로테이션 (주 단위)
──────────────────────────────
DevOps 담당자
각 팀 리드 (백엔드 · 프론트 · 앱 · 엔진)
```

```

```

```
복구 시간   : 2026-03-11 14:52

원인        : DB 커넥션 풀 고갈
Post-mortem : [Confluence 링크]
```

---

### Post-mortem (장애 보고서) 양식

```
Confluence 장애 보고서
──────────────────────────────
1. 장애 개요
   - 발생 시간 / 복구 시간 / 영향 범위

2. 타임라인
   - 시간대별 대응 내역

3. 근본 원인 (Root Cause)
   - 왜 발생했는가

4. 대응 내역
   - 어떻게 해결했는가

5. 재발 방지 대책
   - 무엇을 개선할 것인가
   - 담당자 / 완료 기한
```

## 전체 모니터링 체계 요약

```
클라이언트
├── 앱      → Firebase (Crashlytics · Analytics · Performance)
└── 웹      → Sentry

서버
├── 인프라  → Grafana + Prometheus
├── 로그    → ELK Stack
├── API     → Sentry
└── 업타임  → UptimeRobot

알림
└── 전체 이상 감지 → Slack 연동

장애 대응
├── 등급 분류 (P1~P4)
├── 온콜 체계 (주 단위 로테이션)
└── Post-mortem → Confluence 문서화
```
