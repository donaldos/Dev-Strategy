## 1. Jira (이슈 관리)

### 이슈 타입 정의

| 타입 | 용도 | 예시 |
|------|------|------|
| **Epic** | 큰 기능 단위 | 음성인식 API 개발 |
| **Story** | 사용자 관점 기능 | 사용자가 음성으로 검색할 수 있다 |
| **Task** | 개발 작업 단위 | 음성인식 API 엔드포인트 개발 |
| **Bug** | 버그 수정 | 로그인 토큰 만료 오류 |
| **Hotfix** | 긴급 버그 수정 | Production 결제 오류 |

---

### 이슈 상태 흐름

```
To Do → In Progress → In Review → Done
  │                       │
  │                       └── PR 생성 시 자동 이동
  └── 개발 시작 시 자동 이동
```

---

### Jira ↔ GitHub 연동

브랜치 네이밍에 Jira 이슈 번호를 포함하면 자동 연동됩니다.

```
feature/DEV-123-login-api
bugfix/DEV-456-token-expiry

→ GitHub PR과 Jira 이슈 자동 연결
→ merge 시 Jira 이슈 자동 Done 처리
```

---

### 스프린트 구성

```
스프린트 주기 : 2주
──────────────────────────────
월요일   스프린트 시작 · 계획 미팅
매일     데일리 스탠드업 (15분)
          - 어제 한 일
          - 오늘 할 일
          - 블로커 (막히는 것)
금요일   스프린트 종료 · 회고
```

## 2. Confluence (문서화)

### 문서 구조

```
Confluence
├── 개발 가이드
│   ├── 개발환경 세팅 방법
│   ├── 브랜치 전략
│   ├── 코드 리뷰 규칙
│   └── 배포 프로세스
├── 아키텍처
│   ├── 시스템 구조도
│   ├── API 명세
│   └── DB 스키마
├── 팀별 문서
│   ├── 백엔드
│   ├── 프론트엔드
│   ├── 앱
│   ├── 음성인식 엔진
│   └── DevOps
└── 회의록
    ├── 스프린트 계획
    └── 스프린트 회고
```

---

### 문서화 규칙

```
✅ 반드시 문서화해야 하는 것
──────────────────────────────
API 명세 (신규 · 변경 시)
아키텍처 변경
개발환경 세팅 방법
장애 대응 내역 (Post-mortem)
스프린트 회고

❌ 문서화 불필요
──────────────────────────────
단순 코드 설명 (코드 주석으로 대체)
일회성 메모
```

## 3. Slack (커뮤니케이션)

### 채널 구성

**개발 채널**

| 채널 | 용도 |
|------|------|
| `#dev-backend` | 백엔드 팀 소통 |
| `#dev-frontend` | 프론트엔드 팀 소통 |
| `#dev-app` | 앱 팀 소통 |
| `#dev-engine` | 음성인식 엔진 팀 소통 |
| `#dev-devops` | DevOps 팀 소통 |
| `#dev-qa` | QA 팀 소통 |

**배포 알림 채널 (GitHub Actions 연동)**

| 채널 | 용도 |
|------|------|
| `#deploy-dev` | Dev 배포 알림 |
| `#deploy-staging` | Staging 배포 알림 |
| `#deploy-production` | Production 배포 알림 |
| `#ci-alerts` | 빌드 · 테스트 실패 알림 |

**전체 채널**

| 채널 | 용도 |
|------|------|
| `#general` | 전체 공지 |
| `#random` | 자유 소통 |

---

### Slack 사용 규칙

```
✅ Slack 사용 원칙
──────────────────────────────
1. 중요 결정사항은 Confluence에 문서화
   (Slack 대화는 휘발됨)

2. DM보다 채널 소통 우선
   (팀 전체가 맥락 공유 가능)

3. 긴급 이슈는 @멘션 활용
   @here    → 채널 접속 중인 멤버
   @channel → 채널 전체 멤버

4. 스레드 활용
   → 채널 메시지에 댓글로 달기
   → 채널 노이즈 최소화
```

## 4. Figma (디자인 협업)

### 개발자 ↔ 그래픽팀 협업 흐름

```
그래픽팀 Figma 디자인 완료
      ↓
개발팀에 Figma 링크 공유
      ↓
개발자 Figma에서 직접 스펙 확인
(색상 · 폰트 · 간격 · 이미지 export)
      ↓
구현 후 디자인과 대조 확인
      ↓
불일치 시 Figma 코멘트로 소통
```

---

## 전체 툴 연동 흐름

```
기획팀 → Jira에 이슈 등록
      ↓
개발자 → Jira 이슈 확인 후 브랜치 생성
      ↓
개발 완료 → PR 생성 → GitHub Actions 실행
      ↓
Slack #ci-alerts 알림
      ↓
코드 리뷰 → merge → 자동 배포
      ↓
Slack #deploy-dev / #deploy-staging 알림
      ↓
QA 검증 → Jira 이슈 Done
      ↓
Production 배포 승인 요청
      ↓
Slack #deploy-production 알림
```
